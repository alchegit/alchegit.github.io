# 외부 이미지 생성 서버 방식 비교 보고서

작성일: 2026-07-15

## 결론

서비스형 자동 이미지 생성은 **외부 서버 worker가 공식 이미지 생성 API를 직접 호출하는 방식**이 가장 안정적이다.

다만 현재 목표가 관리자 개인 작업용으로 ChatGPT Pro 구독을 최대한 활용하는 것이라면 첫 단계는 다르게 잡는다. 플랫폼은 웹툰 제작 공정, 컷 구성, 프롬프트 패키지, 이미지 슬롯을 제공하고, 관리자는 본인의 ChatGPT Pro 화면에서 직접 이미지를 생성한 뒤 결과를 가져온다. Codex는 이 흐름을 정리하는 플랫폼과 서버 worker를 만드는 개발 에이전트로 쓴다.

## 전제

- `/webtoon/guide/`는 저장된 샘플 컷과 준비된 시나리오를 보여주는 체험용 페이지다.
- 가이드는 실제 이미지 생성 비용이 없으므로 도토리를 쓰지 않는다.
- 실제 도토리 차감은 외부 서버에서 이미지 생성 작업을 시작할 때만 적용한다.
- ChatGPT Pro에 붙여 넣을 프롬프트 패키지를 만드는 기능은 무료다.
- 관리자가 본인의 ChatGPT Pro 화면에서 직접 생성한 이미지를 플랫폼에 가져오는 흐름은 API 과금이 없다.
- 방문자는 제작 버튼 없이 공개 결과만 본다.
- 서버 접속 문서 기준으로 `EETNA_WEB_HOON`은 Python 3.12와 원격 개발 세팅이 준비되어 있어 worker 운영 후보로 적합하다.
- 서버 접속 비밀번호와 포트 정보는 운영 비밀로 취급하고 저장소에 커밋하지 않는다.

## 방식 A: 외부 서버에서 Codex를 이용하는 방식

### 가능한 구조

1. 유저가 `/webtoon/studio/`에서 `초안 만들기` 또는 `이 컷 다시 그리기`를 누른다.
2. GitHub Pages 프론트엔드는 Supabase Google 로그인 세션을 Oracle API에 전달한다.
3. Oracle API가 세션과 잔액을 검증하고 transaction으로 도토리를 차감한 뒤 `webtoon_generation_jobs`에 job을 만든다.
4. 외부 서버 worker가 Oracle API에서 `ready` 상태 job을 가져온다.
5. Codex 또는 서버 스크립트가 컷별 프롬프트를 정리한다.
6. 최종 이미지는 공식 이미지 생성 API로 생성한다.
7. 생성된 이미지는 Oracle API가 관리하는 외부 서버 정적 파일 경로에 저장한다.
8. worker가 job 상태를 `done`으로 바꾸고 이미지 URL을 기록한다.
9. 프론트엔드는 job 상태를 polling해서 완성 이미지를 보여준다.

### 장점

- 서버에서 실행되므로 API 키를 브라우저에 노출하지 않는다.
- job queue, 실패 재시도, 로그, 비용 추적을 만들기 쉽다.
- Codex가 서버 코드, 프롬프트 템플릿, 이미지 파일 저장 구조, 오류 처리를 계속 개선할 수 있다.
- GitHub Pages 정적 사이트와 잘 맞는다.

### 단점

- Codex 자체를 “이미지 생성 엔진”처럼 쓰면 흐름이 불필요하게 복잡해진다.
- Codex 세션, 로그인, 사용량, 샌드박스 상태를 서비스 트래픽에 맞춰 안정적으로 유지해야 한다.
- 다수 유저 요청을 처리하려면 Codex보다 일반 worker 코드가 더 예측 가능하다.

### 판단

Codex는 이미지 생성 worker를 **개발하고 유지보수하는 도구**로 쓰고, 실제 이미지는 서버 코드가 OpenAI Image API를 직접 호출하는 구조가 좋다.

## 방식 B: ChatGPT 웹을 브라우저로 열어 가상 조작하는 방식

### 가능한 구조

1. 외부 서버나 PC에서 브라우저를 연다.
2. ChatGPT 웹에 로그인한다.
3. 자동화 도구가 프롬프트를 입력한다.
4. 이미지가 생성될 때까지 기다린다.
5. 생성 이미지를 다운로드하거나 화면에서 추출한다.
6. 파일을 서버 저장소로 옮긴다.

### 장점

- API 연동 없이 눈으로 보며 실험하기 쉽다.
- 초기 프로토타입에서 프롬프트 감각을 잡기 좋다.
- 사람이 한두 장을 직접 확인하며 만드는 작업에는 편하다.

### 단점

- 웹 UI 변경, 로그인 만료, 2단계 인증, CAPTCHA, 다운로드 위치 변화에 취약하다.
- 동시 요청 처리와 실패 복구가 어렵다.
- 이미지 결과를 job ID와 안정적으로 연결하기 어렵다.
- 브라우저 세션과 계정 쿠키를 서버에 오래 보관해야 하므로 보안 부담이 크다.
- 서비스 트래픽이 조금만 늘어도 병목이 생긴다.
- OpenAI 약관상 자동/프로그램 방식으로 Output을 추출하는 행위는 문제가 될 수 있으므로 제품 백엔드로 쓰기 부적절하다.

### 판단

ChatGPT 웹 자동화는 **프롬프트 연구용 임시 수동 실험**에는 쓸 수 있지만, 유저가 누르는 서비스 기능으로 연결하지 않는 것이 맞다.

## 권장 아키텍처

### 1단계: 무료 가이드

- `/webtoon/guide/`는 저장된 컷과 시나리오만 재생한다.
- 로그인 없이 실행 가능하다.
- 도토리를 쓰지 않는다.
- 반복 실행 제한을 두지 않는다.

### 2단계: ChatGPT Pro 보조 제작

- 작업대가 컷 구성과 이미지 생성 프롬프트 패키지를 만든다.
- 관리자가 `ChatGPT 프롬프트` 버튼을 눌러 프롬프트를 복사한다.
- 관리자가 본인의 ChatGPT Pro 화면에서 직접 이미지를 생성한다.
- 생성 이미지를 플랫폼 이미지 슬롯에 가져온다.
- 이 흐름은 도토리를 차감하지 않는다.

### 3단계: 실제 생성 job

- `/webtoon/studio/`에서 실제 초안 생성 또는 컷 재생성 버튼을 누를 때만 로그인과 도토리를 요구한다.
- Oracle API가 Google 세션과 도토리를 확인한 뒤 transaction으로 차감하고 job을 만든다.
- 외부 서버 worker가 job을 가져간다.
- 현재 초안 worker는 `workers/webtoon-image-worker/`에 있으며, 기본 `manual-prompt` 모드에서는 OpenAI Image API를 호출하지 않고 ChatGPT Pro용 프롬프트 파일만 만든다.
- 향후 API 생성 모드가 필요해지면 결과 이미지를 Oracle API의 서버 정적 경로에 저장한다.

### 4단계: 운영 안정화

- worker는 systemd 서비스로 실행한다.
- job 실패 시 `failed` 상태와 에러 메시지를 남긴다.
- 429, 5xx 오류는 제한적으로 재시도한다.
- 같은 프롬프트로 너무 자주 요청하는 것을 막는다.
- 생성 이미지는 사용자별/프로젝트별 폴더에 저장한다.
- 오래된 draft 이미지는 정리하는 보관 정책을 둔다.

## 추천 구현 흐름

1. 먼저 `/webtoon/studio/`에 ChatGPT Pro용 프롬프트 패키지와 이미지 슬롯을 만든다.
2. Supabase SQL에서 `webtoon_generation_jobs`와 도토리 RPC를 준비한다.
3. 외부 서버에 `workers/webtoon-image-worker/`를 배치한다.
4. `.env.example`을 `.env`로 복사하고 service role key는 서버 `.env`에만 넣는다.
5. 먼저 `manual-prompt` 모드로 ready job을 prompt outbox 파일로 변환한다.
6. worker는 `ready` job을 가져와 `running`으로 바꾼 뒤, prompt 파일을 만들고 `done`으로 기록한다.
7. 운영자가 prompt를 ChatGPT Pro에 붙여 넣어 이미지를 만들고 `/webtoon/studio/` 이미지 슬롯에 가져온다.
8. 나중에 API 생성 모드를 붙일 때만 OpenAI Image API를 호출한다.
9. API 모드에서는 base64 결과를 이미지 파일로 저장하고 이미지 URL과 metadata를 job에 기록한다.
10. 프론트엔드는 job 상태를 주기적으로 확인하고 결과를 반영한다.

## Codex의 적절한 역할

- 이미지 생성 프롬프트 템플릿 작성
- worker 코드 작성과 테스트
- 실패 로그 분석
- Supabase SQL/RLS 점검
- 생성 결과 품질 평가용 내부 도구 제작
- 오래된 이미지 정리 스크립트 작성

Codex가 매 요청마다 직접 대화형으로 이미지를 생성하는 구조는 피한다.

## 최종 추천

1. 가이드는 도토리 없이 무제한 반복 실행한다.
2. ChatGPT Pro용 프롬프트 패키지 생성도 무료로 제공한다.
3. 유저가 직접 Pro에서 생성한 이미지를 가져오는 수동/반자동 워크플로를 먼저 만든다.
4. 실제 서버 생성 버튼만 도토리를 사용한다.
5. ChatGPT 웹 자동 조작은 핵심 서비스 기능으로 사용하지 않는다.
6. 장기적으로는 외부 서버 worker + 공식 이미지 생성 API를 선택 옵션으로 둔다.
7. Codex는 그 worker와 플랫폼을 만들고 운영하는 개발 보조자로 사용한다.
